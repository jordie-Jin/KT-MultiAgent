# IT 서비스 로그 장애 분석 시스템 구축 제안요청서(RFP)

## 1. 사업 개요

### 1.1 목적

시스템이 쏟아내는 운영 로그를 입력받아 이상과 장애 징후를 가려내고, 어떤 로그가 왜 문제인지 근거와 함께 설명하는 시스템을 구축한다. 분석 결과를 영속적으로 보관하여 사후 조회와 점검 이력으로 활용할 수 있게 한다.

### 1.2 배경

- 서비스가 정상 운영 중에도 로그는 끊임없이 쌓인다. 그 안에 섞인 장애 징후는 전체에서 소수에 해당하므로, 사람이 로그를 일일이 훑어 이상을 제때 찾기는 어렵다.
- 장애 징후를 늦게 발견할수록 영향 범위가 커진다. 사후에 알아차리면 이미 서비스 중단이나 사용자 피해가 발생한 경우가 많다.
- 어떤 로그를 이상으로 본 이유나 평소와 달랐던 패턴이 기록으로 남지 않으면, 같은 장애가 다시 나도 원인을 추적하거나 담당자 간 공유하기 어렵다.
- 정상과 이상만 가르는 도구는 결과만 제시하므로, 담당자가 조치하려면 어떤 로그가 왜 문제인지의 근거가 별도로 필요하다.

### 1.3 과업 범위

- 운영 로그를 입력받아 정상과 이상(장애 징후)을 판정하고, 어떤 로그가 왜 문제인지 근거를 제시하는 기능을 구축한다.
- 분석 결과를 영속적으로 보관하고 사후에 조회할 수 있는 서비스를 구축한다.
- 운영 시스템 제어나 자동 복구 실행은 범위에서 제외한다. 로그 분석, 판정, 조회에 한정하며 실제 서버나 서비스를 조작하지 않는다.

## 2. 요구사항

### 2.1 필수 요구사항

- 운영 로그를 입력받아 정상 로그와 이상(장애 징후) 로그를 일관된 기준으로 구분할 수 있어야 한다.
- 이상으로 판정한 로그에 근거를 붙여 제시할 수 있어야 한다. 근거는 평소 패턴을 벗어난 항목을 지목하거나, 어떤 로그가 왜 문제인지를 사람이 읽을 수 있는 형태로 설명한다.
- 반복해서 나타나는 이상 패턴(원인 패턴)을 묶어 제시할 수 있어야 한다. 같은 성격의 장애 징후를 한데 모아 보여 주면 충족한다.
- 분석 결과를 영속적으로 보관하고, 사용자가 사후에 조회할 수 있는 서비스 형태로 제공해야 한다. 결과를 저장한 뒤 목록과 상세로 다시 조회할 수 있으면 충족한다.
- 라벨이 포함된 로그(정상과 알람 구분)로 판정 정확도를 직접 확인할 수 있어야 한다.

### 2.2 확장(우대) 요구사항

확장 요구사항은 우대 항목이며 필수 요구사항 충족 이후 적용한다.

- 정상으로 분리된 로그와 이상으로 분리된 로그를 모두 활용해, 정상 기준을 학습한 뒤 그 기준을 벗어난 로그를 이상으로 가려낼 수 있다.
- 빠른 1차 탐지(규칙, 통계 기준)와 원인 근거 설명의 책임을 역할로 분리하여 처리할 수 있다. 분리 단위는 수행팀이 정한다.
- 판정 규칙을 코드에 고정하지 않고 에이전트가 호출하는 도구(tool) 형태로 구성하고, 에이전트가 도구 결과를 받아 장애 원인을 설명하는 추론과 결합할 수 있다.
- 1차 탐지와 원인 설명을 다수 에이전트로 분리해 협업하게 하고, 일부 처리가 실패해도 전체가 중단되지 않도록 처리할 수 있다.
- 저장된 로그를 시간 순서대로 다시 흘려보내(리플레이) 들어오는 로그를 실시간처럼 감시하고, 이상 징후가 나타나면 그 시점에 알릴 수 있다.
- 로그인을 두어 허가된 사용자만 분석 결과를 조회하도록 통제할 수 있다.

### 2.3 비기능 요구사항

- 정형 또는 반정형 로그를 대상으로 하며, 실서비스를 실시간으로 직접 감시하는 형태는 요구하지 않는다.
- 단순 RAG 챗봇 형태가 아닌 실제 과제형 시스템으로 구축한다.
- 실시간 감시 요구는 저장된 로그를 시간 순서대로 다시 흘려보내는 리플레이 시뮬레이션에 한정하며, 이를 산출물에 정직하게 기술한다.
- 운영 시스템 제어나 자동 복구 실행 기능은 사용하지 않는다. 분석, 판정, 조회에 한정한다.
- 데이터는 회원가입이나 본인인증이 필요 없는 무인증 공개 데이터에 한정한다.

## 3. 제공 데이터

### 3.1 기본 제공 데이터

| 데이터 | 출처 | 성격 | 접근 방법 |
|--------|------|------|----------|
| Loghub BGL | raw.githubusercontent.com/logpai/loghub | 슈퍼컴퓨터 시스템 로그 2,000라인, 각 라인 1열에 정상과 알람을 구분하는 인라인 라벨 포함 | 무인증, 직접 다운로드 |

- 각 라인 1열의 라벨이 `-`이면 정상, 그 외 값이면 알람 코드에 해당한다. 이 라벨로 정상과 이상을 구분하는 기준을 세우고 판정 정확도를 검증할 수 있다.
- 데이터 적재 예시:

```bash
curl -L -o BGL_2k.log "https://raw.githubusercontent.com/logpai/loghub/master/BGL/BGL_2k.log"
```

- 라인 1열의 라벨을 분리한 뒤 나머지 본문을 분석 대상으로 삼는다.
- 사용할 컬럼, 전처리 방식, 검증 묶음 구성은 수행팀이 정한다.

### 3.2 대체 데이터

기본 제공 데이터의 다운로드가 실패하거나 사내망에서 차단되는 경우 아래 대체 경로를 사용한다. 대체 데이터를 사용해도 이상 탐지, 근거 제시, 보관, 조회의 핵심 흐름은 동일하며 데이터 출처만 변경된다.

| 대체 데이터 | 접근 방법 | 적용 시점 | 유의 사항 |
|------------|----------|----------|----------|
| OpenStack 전체 로그 | zenodo.org/api/records/8196385/files/OpenStack.tar.gz/content | 정상과 이상이 분리된 라벨 데이터로 정상 기준 학습 흐름까지 검증할 때 | 정상 로그와 이상 로그가 별도 파일로 분리되어 있으며 압축 용량이 크므로 사전 확인 필요 |
| Loghub HDFS 구조화 CSV | raw.githubusercontent.com/logpai/loghub/master/HDFS/HDFS_2k.log_structured.csv | 인라인 라벨 파싱 없이 컬럼이 나뉜 구조화 형태로 바로 분석할 때 | 이미 컬럼으로 분리된 구조이므로 라벨 구성과 컬럼명을 사전 확인 필요 |

- 대체 데이터마다 라벨이 들어 있는 위치(인라인, 별도 파일, 컬럼)가 다르므로, 정상과 이상을 구분하는 라벨을 먼저 확인한 뒤 사용한다.
- 대체 데이터를 사용하는 경우에도 운영 시스템 제어는 사용하지 않으며 분석, 판정, 조회 범위는 유지한다.

| 산출물 | 형식 | 제출 방식 |
|--------|------|----------|
| 실행 가능한 소스코드 | GitHub 저장소(로컬 재구동 README 포함) | 저장소 링크 제출 |
| 설계 설명서 | 시스템 구성, 설계 판단 근거, 구현 범위를 기술 | 저장소 내 문서 또는 별도 파일 |
| 시연 자료 | 로그 입력 시 이상 판정과 원인 근거 산출, 결과 조회 장면 포함 | 시연 |

- README에는 실행 환경과 의존성, 데이터 확보 방법, 구동 절차, 주요 화면 또는 진입점을 포함한다.
- 확장 요구사항을 수행한 경우 리플레이 감시 장면이나 협업 분석 장면을 시연에 함께 포함한다.

## 5. 평가 기준

- 필수 요구사항(2.1)의 전 항목이 실제로 동작하는지를 기준선으로 평가한다.
- 확장 요구사항(2.2) 충족 항목은 가산하여 평가한다.
- 이상 로그 판정의 정의와 원인 근거 항목 설계의 적절성을 비중 있게 평가한다.
- 라벨 데이터를 활용한 판정 정확도 검증 결과를 평가에 반영한다. 정확도 100퍼센트가 아니더라도 어떤 로그가 왜 문제인지의 근거가 명확히 제시되는지를 우선한다.

## 6. 수행 환경 및 제약

### 6.1 기술 환경

- Python 기반 에이전트와 Spring Boot 백엔드를 사용한다.
- 시스템 아키텍처, 에이전트 구성, 데이터 스키마, 세부 기술 선택, 구현 방식은 수행팀이 설계한다. 본 RFP는 정답 설계를 제시하지 않으며, 요구 능력과 경계만 정의한다.
- 사용할 LLM은 수행팀이 정한다. 문서나 발표에 모델명이나 가격을 기재할 때는 최신 기준으로 확인한다.

### 6.2 제약

- 운영 시스템 제어나 자동 복구 실행은 범위에서 제외한다. 로그의 분석, 판정, 조회에 한정한다.
- 실시간 감시 요구는 저장된 로그의 리플레이 시뮬레이션에 한정하며 산출물에 정직하게 기술한다.
- 데이터는 무인증 공개 데이터에 한정한다.

## 부록. 용어 정의

| 용어 | 정의 |
|------|------|
| 운영 로그 | 시스템이나 서비스가 동작하면서 시간 순서로 남기는 기록 |
| 장애 징후(이상 로그) | 정상 운영 패턴에서 벗어나 장애나 오류가 의심되는 로그 |
| 원인 패턴 | 반복해서 나타나는 이상 로그를 성격별로 묶은 것 |
| 리플레이 | 저장된 로그를 시간 순서대로 다시 흘려보내 실시간 유입처럼 감시하는 방식 |
| 에이전트 도구(tool) | 에이전트가 호출하여 탐지 규칙 등을 실행하고 결과를 받는 외부 기능 단위 |
| 서비스 형태 | 분석 결과를 파일 또는 DB에 보관하고 목록과 상세로 재조회할 수 있는 형태 |
